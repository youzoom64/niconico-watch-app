# GUI First AI Workflow

AIにGUIアプリを作らせる時の指示テンプレート。

目的は、ユーザーがPCの前に張り付かなくても済むように、GUIを先に完成形として固定し、そのGUIを実装契約として使うこと。

## 基本方針

- GUIを先に作る。
- GUIを最終仕様、操作契約、実装契約として扱う。
- ユーザーがGUIを承認するまで、バックエンド実装へ進まない。
- GUIにない機能、API、状態、裏処理を勝手に作らない。
- 配線は1本ずつ実装し、必ずGUI上で操作確認してから次へ進む。
- 報告は事実だけにする。慰め、謝罪、励まし、人格評価は禁止。

## 最初に投げる短い指示

```text
GUI先行で作れ。
GUIを最終仕様として扱い、承認までバックエンド実装禁止。
承認後、全UI部品に action_id / state_key / data_source / error_target / dangerous_operation を割り当てて UI_CONTRACT.md を作れ。
その契約にない機能・API・画面は作るな。
配線は1本ずつ実装し、必ずGUI上で操作確認してから次へ進め。
報告は事実だけ。慰め・謝罪・励ましは禁止。
```

## 詳細指示テンプレート

```text
目的:
GUIを先に完成形として作る。
このGUIを最終仕様・操作契約・実装契約として扱う。

禁止:
- バックエンド実装を先に進めるな
- GUIにない機能を作るな
- GUIにないAPIを作るな
- ユーザー確認なしに画面構造を変えるな
- 「仮UI」「あとで直すUI」を完了扱いするな

Phase 1:
GUIだけを作る。
全ボタン、入力欄、一覧、状態表示、ログ表示、エラー表示を配置する。
この時点では実処理は全部ダミーでよい。

Phase 1成果物:
- 実際に起動できるGUI
- スクリーンショット
- UI部品一覧
- 未配線一覧

Phase 1完了条件:
ユーザーがGUIを見て承認すること。
承認なしにPhase 2へ進むな。

Phase 2:
GUIを契約書化する。
各UI部品に以下を割り当てる。
- action_id
- state_key
- data_source
- error_target
- dangerous_operation true/false

Phase 2成果物:
- UI_CONTRACT.md
- action/state/data_source対応表

Phase 3:
配線表を作る。
まだ実装するな。

各action_idについて:
- 入力
- 出力
- DB操作
- 外部コマンド
- 成功表示
- 失敗表示
- ロック条件
- テスト方法
を書く。

Phase 4:
配線を1本ずつ実装する。
1本ごとにGUI上で実操作して確認する。
画面で確認できない実装は完了扱いしない。

報告ルール:
- できた/未達/次に必要な確認だけ返す
- 慰め、謝罪、励まし、人格評価は禁止
- 判断が必要な時だけユーザーへ戻す
```

## AIへの運用制約

- GUI承認前にDB、API、外部コマンド、投稿、録画、AI要約などを実装しない。
- GUIにない裏機能を「気を利かせて」追加しない。
- 未配線のUIは未配線として明示する。
- 実装済み、ダミー、ロック中、未実装を画面上で区別する。
- 危険操作は必ず `dangerous_operation` として明示する。
- 危険操作はユーザーの明示承認なしに実行しない。
- 完了報告は、GUI上で実際に押して確認したものだけに限定する。

## チェックリスト

- GUIはユーザーが承認した形か。
- すべてのボタンに `action_id` があるか。
- すべての表示欄に `state_key` があるか。
- すべての一覧に `data_source` があるか。
- すべてのエラー表示に `error_target` があるか。
- 危険操作に `dangerous_operation` が付いているか。
- GUIにないAPIを作っていないか。
- GUIにない状態を返していないか。
- 1本ずつ配線し、GUI上で操作確認したか。
- 未達を完了扱いしていないか。
